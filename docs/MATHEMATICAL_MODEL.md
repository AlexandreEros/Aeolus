# Mathematical Model — As Implemented

This document reconstructs, from the source code as of branch `geodesic-grid-refactor`
(commit `2a30a18`), the governing equations and the numerical method that are *actually
implemented* — including places where the implementation deviates from its own docstrings.
Deviations and defects are cross-referenced to [KNOWN_RISKS.md](KNOWN_RISKS.md) by ID (R-1, R-2, …).

All statements marked **[measured]** were verified numerically during the audit
(2026-07-11) on the repository venv (Python 3.12, CuPy 13.4, NVIDIA GeForce MX110).

---

## 1. Governing equation

The dynamical core intends to solve the **barotropic vorticity equation (BVE)** on a
rotating sphere ([barotropic_vorticity.py](../src/planetary_sandbox/run/bve/barotropic_vorticity.py)):

```
∂ζ/∂t + J(ψ, ζ + f) = ν ∇²ζ + F
∇²ψ = ζ
u = k × ∇ψ
```

with

- `ζ` — relative vorticity (radial component of curl of the horizontal wind),
- `ψ` — streamfunction,
- `f = 2Ω sin(φ)` — planetary vorticity (φ = latitude), correctly built from latitude in
  `BarotropicVorticity.__init__` (the code comment about colatitude is stale),
- `ν` — constant kinematic viscosity acting through the exact spectral Laplacian,
- `F` — optional spectral forcing (always `None`/zero in the current runner),
- `J(a,b) = (1/(R² cos φ)) (∂a/∂λ ∂b/∂φ − ∂a/∂φ ∂b/∂λ)` — the spherical Jacobian, so that
  `J(ψ, q) = u·∇q`.

**What is actually integrated differs from this.** The tendency routine evaluates the
Jacobian through `SpectralOperators.jacobian_pseudospectral`, which algebraically reduces to
`−cos φ · J(ψ, η)` instead of `J(ψ, η)` (wrong sign *and* a spurious cos φ amplitude factor).
The advection term entering the model is therefore `+cos φ · J(ψ, η)` rather than
`−J(ψ, η)`. **[measured]**: advecting single spherical-harmonic modes with an exact
solid-body streamfunction yields ratios of −0.76 to −0.81 relative to the analytic result,
while the unused velocity-form operator `advect_scalar_by_streamfunction` reproduces the
analytic result to 4–5 digits. See **R-1** — this is the highest-severity finding of the audit.

There is no topography, no divergence, no height field, and no bottom drag: the model is a
single-level, non-divergent flow on a perfect sphere. Terrain generated by `Planet.generate`
is decorative and does not enter the dynamics.

### 1.1 Prognostic and diagnostic variables

| Variable | Role | Representation |
|---|---|---|
| `ζ_lm` | **Prognostic** (the only one) | complex spectral coefficients, dense `(l_max+1, l_max+1)` CuPy array, m ≥ 0 |
| `ψ_lm` | Diagnostic | `ζ_lm / λ_l`, `λ_l = −l(l+1)/R²`, l = 0 mode zeroed |
| `u, v` | Diagnostic | east/north velocity on grid points, from spectral derivatives of ψ |
| `η = ζ + f` | Diagnostic | formed **on the grid** each tendency call, then re-transformed (a lossy round trip; see R-2/R-5) |

## 2. Spherical-harmonic conventions

### 2.1 Basis and normalization

All engines use **fully normalized (orthonormal) complex spherical harmonics with the
Condon–Shortley phase**:

```
Y_l^m(θ, λ) = N_l^m · P_l^m(cos θ) · e^{imλ},      N_l^m = sqrt( (2l+1)/(4π) · (l−m)!/(l+m)! )
```

- `θ` = **colatitude** (0 at north pole), `λ` = longitude, m ≥ 0 only.
- The Condon–Shortley factor `(−1)^m` is folded into the associated Legendre functions
  inside the CUDA kernels ([legendre.cu](../src/planetary_sandbox/numerics/cuda/legendre.cu),
  [sh_matrix.cu](../src/planetary_sandbox/numerics/cuda/sh_matrix.cu)) via the seed
  `P_m^m = (−1)^m (2m−1)!! (1−x²)^{m/2}`; the Python-side normalization deliberately does
  *not* apply it a second time (the "FIX" comments in
  [spherical_harmonics.py](../src/planetary_sandbox/numerics/spherical_harmonics.py) record a
  past double-phase bug).
- Orthonormality means `∫ Y_l^m (Y_{l'}^{m'})* dΩ = δ_ll' δ_mm'`, so a constant field maps to
  `a_00 = sqrt(4π) · c`. **[measured]**: the geodesic transform reproduces `a_00 = sqrt(4π)`
  and `a_10 = sqrt(4π/3)` for `f = 1` and `f = sin φ` to 15 digits (resolution 3, l_max 5).
- Degree recursion in both kernels is the standard
  `P_l^m = [(2l−1)x P_{l−1}^m − (l+m−1)P_{l−2}^m]/(l−m)`; normalization uses `lgamma` to
  avoid factorial overflow. Note the *normalization is applied after the unnormalized
  recursion*, so very high degrees (l ≳ 150) would overflow before normalizing; irrelevant
  at current l_max ≤ 45 but a known scaling limit.

`sh_matrix_real.cu` (a real-basis variant) exists but is **never loaded** by any Python code.

### 2.2 Spectral coefficient layout

Two layouts coexist, with fixed mappings between them:

1. **Dense grid layout** (public API): complex array `a[l, m]` of shape
   `(l_max+1, l_max+1)`; entries with `m > l` are ignored (and must be treated as garbage,
   not zeros — some producers write into them, e.g. `_random_low_l`).
2. **Flat basis layout** (internal to `PointSetSphericalHarmonics`): column index
   `k = l(l+1)/2 + m` of the basis matrix `Y ∈ ℂ^{n_points × n_basis}`,
   `n_basis = (l_max+1)(l_max+2)/2`, ordered `(0,0), (1,0), (1,1), (2,0), …`.

Negative-m coefficients are never stored; real fields are assumed, so
`a_{l,−m} = (−1)^m a_{l,m}*` is implicit in the synthesis formula:

```
f(x_i) = Σ_l a_{l,0} Y_{l,0}(x_i) + 2 Re Σ_{l, m>0} a_{l,m} Y_{l,m}(x_i)
```

implemented in `inv_transform` as `2·Re(Y a) − Re(Y a|_{m=0})`. Consequence: **the
imaginary part of m = 0 coefficients is silently discarded**, and complex fields cannot be
represented.

## 3. Grids

### 3.1 Geodesic grid (operational)

[geodesic_grid.py](../src/planetary_sandbox/numerics/geodesic_grid.py): icosahedron subdivided
`resolution` times (each level splits every triangle in four; vertices projected to the
sphere), giving `n_points = 10·4^res + 2` (642 at res 3, 2562 at res 4, 10242 at res 5).
The base icosahedron is **rotated by 0.01 rad about x** ("anti-singularity rotation") so no
vertex lands exactly on a pole; this is what keeps the `1/cos φ` factors in the operators
finite. Per-vertex quadrature weights are Voronoi-style: each spherical triangle
(L'Huilier's formula) contributes one third of its area to each of its vertices. Weights are
converted to solid angle and renormalized to `Σw = 4π` in the SH wrapper.
An inverse-distance-weighted adjacency matrix and `min_edge_length` support the (currently
latent) local finite-difference operators and the CFL estimate.

### 3.2 Latitude–longitude grid (legacy baseline)

[grid.py](../src/planetary_sandbox/numerics/grid.py): uniform colatitude including both poles,
uniform longitude with `endpoint=False`; forced to odd point counts for Simpson quadrature.
Used only by tests/diagnostic scripts and the interpolation utilities, not by the BVE.

## 4. Grid transforms

### 4.1 Point-set / geodesic transform (operational path)

`PointSetSphericalHarmonics` precomputes the dense basis matrix `Y` on the GPU
(one CUDA kernel launch), then:

- **Analysis (forward)**: `a = Yᴴ (w ⊙ f)` — discrete quadrature approximation of
  `∫ f Y* dΩ` with weights `w` (default: normalized Voronoi areas).
- **Synthesis (inverse)**: dense mat-vec, formula in §2.2.

This transform pair is **not exact**: analysis∘synthesis equals identity only if the Gram
matrix `G = Yᴴ diag(w) Y = I`. **[measured]** round-trip quality with Voronoi weights:

| config | pts/basis | worst diagonal error | worst leakage (rms of spurious coeffs) |
|---|---|---|---|
| res 4, l_max 15 | 18.8 | 2.2e−3 (l=15) | 6e−3 |
| **res 4, l_max 21 (CLI default)** | **11.1** | ≈2e−3 (l=21) | ≈1e−2 |
| res 4, l_max 45 (audit worst case) | 2.4 | 7.1e−2 (l=45) | 3.0e−1 |
| res 5, l_max 45 | 9.5 | 9.0e−4 | 1.0e−2 |

The **default CLI configuration (`--l-max 21 --resolution 4`)** keeps
approximately eleven grid points per SH basis function; the historical
l_max = 45 default (recorded in the audit as severely under-resolved,
R-2) has been retired. Low degrees (l ≤ 2) are exact to machine precision
at every tested resolution.

`GeodesicSphericalHarmonics` wraps this with three weight modes: `"voronoi"` (default),
`"uniform"`, and `"optimize"` (bounded-variable least squares solving `Σ_i w_i |Y_ki|² = 1`
for every basis function plus `Σw = 4π`, cached to a pickle keyed by resolution/l_max/point
count — but not by radius or by grid rotation; see R-13).

### 4.2 Lat–lon transform (legacy)

`LatLonSphericalHarmonics` computes coefficients mode-by-mode by 2-D Simpson quadrature of
`f · Y* · sin θ`. **[measured]** the longitude axis is built with `endpoint=False` while
Simpson's rule integrates only over `[λ_0, λ_{N−1}]`, so **the last longitude panel is
dropped: every coefficient is low by exactly a factor `(1 − 1/n_lon)`** (a_00 = 3.4904 vs
√(4π) = 3.5449 at n_lon = 65, deficit = 1/65). Additional spurious coefficients O(3e−2)
appear because Simpson weights are inappropriate for a periodic open grid. See R-7. This
engine is used as the reference in two test scripts, which weakens those tests.

### 4.3 Grid-to-grid interpolation

[grid_interpolation.py](../src/planetary_sandbox/numerics/grid_interpolation.py) round-trips
geodesic ↔ lat–lon via `scipy.interpolate.griddata` (CPU). Used for plotting only.

## 5. Derivative operators (spectral)

Implemented in [spectral_operators.py](../src/planetary_sandbox/numerics/spectral_operators.py),
all acting on the dense (l, m) coefficient array:

- **Laplacian**: diagonal, `∇²Y_l^m = −l(l+1)/R² · Y_l^m`. Exact.
- **Inverse Laplacian**: division by eigenvalues; the l = 0 mode is assigned the arbitrary
  eigenvalue `1/R²` in `SpectralOperators.inv_laplacian` (documented as arbitrary) and is
  *zeroed* in `BarotropicVorticity.vorticity_to_streamfunction`. Both are harmless because
  derivatives annihilate constants, but the two code paths disagree.
- **Zonal derivative**: `(∂/∂λ)Y_l^m = im·Y_l^m`, returned pre-divided by R
  (`d_lambda_coeffs = (im/R)·a`). Note: the operator *mutates its own precomputed buffer* on
  every call to zero the m = 0 entry (R-11, cosmetic).
- **Meridional operator**: `sin θ ∂/∂θ` via the standard three-term recurrence for
  orthonormal harmonics,

  ```
  sinθ ∂Y_l^m/∂θ = l·ε_{l+1,m} Y_{l+1}^m − (l+1)·ε_{l,m} Y_{l−1}^m,
  ε_{l,m} = sqrt( (l²−m²) / (4l²−1) )
  ```

  implemented with correct band coupling (audit re-derived and spot-checked the l = 1,
  m = 0 identity analytically). Note `sin θ ∂/∂θ = −cos φ ∂/∂φ`.
- **Velocity from streamfunction** (`velocity_from_streamfunction`):
  `u = (1/(R cosφ))·(sinθ ψ_θ) = −ψ_φ/R`, `v = (ψ_λ/R)/cosφ`. **[measured]** exact to 9e−15
  for solid-body rotation. Points with |cos φ| < 1e−6 are set to NaN then zeroed (never
  triggered thanks to the grid rotation).
- **Scalar gradient** (`grad_from_scalar`): `(q_λ/(R cosφ), q_φ/R)` — correct, but divides
  by `grid.coslat` with no clamping (R-12).
- **Jacobian** (`jacobian_pseudospectral`): computes
  `J_code = (a_λ · sinθ b_θ − sinθ a_θ · b_λ) / (R² cos φ)`. Substituting
  `sinθ ∂θ = −cosφ ∂φ` gives **`J_code = −cos φ · J(a,b)`**: the intended `1/cos²φ` is a
  `1/cos φ`, and the sign is flipped. This is the operator the BVE tendency uses (R-1).
- **Velocity-form advection** (`advect_scalar_by_streamfunction`): `u·∇q` from the two
  verified operators above. **[measured] correct** — but currently *unused* by the model.

A completely separate **local (non-spectral) operator family**,
[differential_operators_spherical.py](../src/planetary_sandbox/numerics/differential_operators_spherical.py),
builds sparse least-squares gradient/divergence/curl matrices from the mesh adjacency and a
**graph Laplacian** (`D − W`, which is *not* a consistent discretization of the surface
Laplacian — no area/metric scaling). It is instantiated lazily by `SpectralOperators
._get_differential_ops` but **no live code path calls it**; treat it as latent/experimental
(R-16).

## 6. Nonlinear products and dealiasing

Nonlinear terms are formed pseudospectrally: derivative fields are synthesized on the
geodesic points, multiplied pointwise, and re-analyzed. "Dealiasing" is a post-hoc
truncation: after transforming the product, coefficients with `l > ⌊2·l_max/3⌋` (and,
redundantly, `m > ⌊2·l_max/3⌋`) are zeroed.

This is **not equivalent to the classical 2/3 rule**: there is no finer intermediate grid,
and the quadrature (Voronoi weights on a geodesic point set) does not integrate products of
two degree-l_max fields exactly, so aliased energy contaminates the *retained* modes before
truncation. The truncation only removes the modes above the cutoff; it cannot undo
in-band aliasing (R-3). The quantitative aliasing behaviour of this transform has not been
characterized — that is a Stage-A validation task.

Additional inefficiency: `tendency()` calls the Jacobian with `dealias=True`, which
transforms the product to spectral space, truncates, synthesizes back to the grid — and then
`tendency()` immediately re-analyzes that grid field to spectral space. Two extra transforms
per evaluation, eight per RK4 step.

## 7. Time integration

- **Scheme**: classical explicit **RK4** on the spectral vorticity coefficients
  ([runner.py](../src/planetary_sandbox/run/bve/runner.py) `rk4_step`), diffusion included
  explicitly (no integrating factor / implicit treatment).
- **Step-size policy**: a **state-adaptive advective CFL ceiling**. The ceiling
  `dt_cfl = 0.5 · cfl_length_scale / max|u|` (falls back to 600 s if degenerate; the sole
  implementation is `advective_cfl_timestep` in
  [config.py](../src/planetary_sandbox/run/bve/config.py)) is formed from the *initial*
  state for the first step and then **recomputed from every accepted state** — the runner
  drives it with the `max_speed_ms` already produced by the per-step diagnostics record, so
  no extra velocity reconstruction is added. A run in which the maximum speed grows (a smoke
  run went 25.4 → 37.5 m/s) therefore tightens `dt` as the flow accelerates instead of
  eroding the margin (R-4, now the advective condition is controlled). This is *only* the
  advective CFL condition: it is **not** embedded-error (RK45) adaptivity, and it does **not**
  bound RK4 stability for the explicit `ν∇²` diffusion term. The mesh edge length is also not
  the natural CFL length for a spectral method (the resolved wavelength `2πR/l_max` is) — a
  separate open item.
- **Output-time clipping**: canonical count mode shortens individual steps to
  `min(dt_cfl, next_scheduled_time − t, t_end − t)` and integrates every positive residual,
  so it lands exactly on every requested output time and on `t_end`; a step that cannot
  advance representable time aborts rather than being enlarged past `dt_cfl`. Legacy
  interval mode intentionally follows the historical countdown instead and may stop within
  `1e-6 · dt_snapshots` of a misaligned `t_end`; that path preserves main's storage boundaries
  and stopping semantics rather than adopting count mode's exact-final-time contract. The
  accepted-step sequence now varies with the evolving flow speed in both modes; the snapshot
  and stopping contracts are what "legacy preservation" refers to, not a frozen step sequence.
  The scheduling seam is the incremental `IntegrationScheduler` (one `next_event(dt_cfl)`
  per accepted step); `integration_plan` remains as a constant-ceiling compatibility/test
  helper.
- **Leapfrog**: `step_leapfrog` exists but is dead code and would crash if called
  (it passes raw arrays where a `BarotropicState` is asserted) (R-6).
- The l = 0 row of every tendency is hard-zeroed ("mass conservation"), which pins the
  global circulation `∮ζ dA` to its initial value. **[measured]** circulation stays at
  round-off (≈5e−8 m²/s absolute) over 60 steps.

## 8. Filtering and dissipation

- **Explicit**: `ν ∇²ζ` with the true spectral eigenvalues (the `laplacian_eig[0] = 1/R²`
  hack would inject spurious l = 0 growth, but the hard-zeroing of the l = 0 tendency masks
  it). Default `ν = 0`. No hyperviscosity option.
- **Truncation**: the 2/3-style zeroing described in §6, every tendency evaluation.
- **Implicit / unintended**: every tendency evaluation round-trips `η = ζ + f` through
  analysis∘synthesis (grid-space addition of f), which acts as a lossy filter whenever the
  Gram matrix ≠ I — i.e. always, and severely at any under-sampled l_max/resolution pairing.
  **[measured]** with ν = 0: kinetic energy −2.4 %, enstrophy −1.4 % over ≈6 simulated days
  at res 4 / l_max 20; kinetic energy −2.5 % in *12 simulated hours* at the historical
  res 4 / l_max 45 pairing (R-2, R-5), which is what motivated shipping l_max 21 as the CLI
  default. A spectrally clean BVE core should hold these invariants to time-discretization
  accuracy (orders of magnitude tighter).

## 9. Coordinate and sign conventions

| Quantity | Convention |
|---|---|
| Longitude λ | geodesic grid: `atan2(y,x) ∈ [−π, π]`; lat–lon grid: `[0, 2π)`; both accepted by the basis kernels (only `e^{imλ}` matters) |
| Latitude φ | geocentric, `[−π/2, π/2]`, `atan2(z, √(x²+y²))` |
| Colatitude θ | `π/2 − φ`, used inside kernels and the meridional operator |
| Velocity | `u` east, `v` north (m/s) |
| Coriolis | `f = 2Ω sin φ`, `Ω = 2π/sidereal_day`; **CLI default `--day-hours inf` ⇒ Ω = 0 (non-rotating!)** — rotation must be requested explicitly (R-9) |
| Radius | dynamics use `params.radius` = volumetric mean radius of the oblate spheroid; the grid is also built with that radius; the vorticity viewer instead uses `equatorial_radius` for its area weights (inconsistent, cosmetic — R-14) |
| Sphere | dynamics are on a perfect sphere; oblateness affects only decorative terrain/`radial_distances` |
| Units | SI throughout; ζ in s⁻¹, ψ in m²/s |

## 10. CPU and GPU execution paths

**There is no CPU path.** Every numerics module imports CuPy at module scope and the basis
matrices, states, and operators live in GPU memory; `requirements.txt` pins
`cupy-cuda11x`. NumPy/SciPy appear only for mesh construction, weight optimization
(`scipy.optimize.lsq_linear` on CPU), plotting interpolation, and I/O. Consequences:

- The package cannot be imported, and no test can run, without a CUDA GPU (R-15) —
  including on typical CI runners.
- "CPU/GPU numerical agreement" (Stage A, test A-9 in
  [VALIDATION_PLAN.md](VALIDATION_PLAN.md)) currently has no CPU side to compare against;
  the plan proposes a test-only NumPy/SciPy reference basis for exactly this reason.

GPU specifics:

- Custom CUDA kernels: `legendre.cu` (per-mode P_l^m), `sph_harm.cu` (per-mode Y from
  precomputed P), `sh_matrix.cu` (whole basis matrix, the operational one),
  `sh_matrix_real.cu` (unused). All float64/complex128.
- After basis construction, transforms are cuBLAS `zgemv`/`zgemm` calls; the BVE step is
  dominated by these. **[measured]** ≈41 ms per RK4 step at res 4 / l_max 20 on a GeForce
  MX110 (2 GB).
- Memory scales as `16·n_points·(l_max+1)(l_max+2)/2` bytes for `Y` (res 6 with l_max 63
  would already exceed 2 GB); dense-matrix transforms are O(N·L²) work versus
  O(N·L) for a Gauss–Legendre/FFT transform — this is the main scalability ceiling.
- **[observed]** on Windows with the display GPU, long-running kernels/GEMMs can hit the
  WDDM TDR watchdog: the audit's res-5 Gram-matrix computation died with
  `cudaErrorLaunchTimeout`. Production runs at higher resolution need a compute-dedicated
  GPU or a raised TDR delay (R-15).

## 11. Summary of what is solid vs. suspect

Verified correct in this audit (**[measured]**):

- Basis normalization/orthogonality at adequate sampling (Gram ≈ I to 1e−3 at ≥9 pts/basis);
- Laplacian and its inverse; zonal derivative; meridional `sinθ ∂θ` operator;
- velocity-from-streamfunction and scalar gradient; velocity-form advection `u·∇q`;
- circulation pinning; RK4 implementation as written.

Demonstrably wrong or unquantified (details in [KNOWN_RISKS.md](KNOWN_RISKS.md)):

- the Jacobian actually used by the tendency (R-1);
- the default resolution/truncation pairing (R-2);
- the dealiasing claim (R-3); the advective CFL is now state-adaptive (R-4 fixed for the
  advective condition; explicit-viscosity stability still uncontrolled); invariant drift (R-5);
- the legacy lat–lon transform's quadrature (R-7);
- everything downstream of these (all published imagery in `out/` was produced with R-1
  active and therefore does not depict solutions of the BVE).
